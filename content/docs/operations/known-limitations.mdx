title: Known Limitations
description: Current product limitations and recommended workarounds.

This page lists user-facing constraints in the current Thyme release. Each item has a recommended workaround.

## Min and Max aggregations

The `Min` and `Max` aggregation operators are **not currently recommended for production**. Under load, these operators may produce incorrect values for some entities.

**Workaround:** Use `Count`, `Sum`, `Avg`, or `ApproxPercentile` instead. All four are invertible operators and have been validated under sustained production workloads (see the [case studies](/docs/case-studies/fraud-detection) for benchmarks).

## Backfill jobs for streaming sources

Defining a streaming-sourced dataset (e.g. Kinesis or Kafka) and triggering a backfill is not currently supported and may cause the engine to fail to start the backfill.

**Workaround:** For streaming sources, rely on the live ingestion path; do not request a backfill on commit. Polling sources (Postgres, Iceberg, S3, BigQuery, Snowflake) backfill normally.

## Engine rebalancing on horizontal scale-up

When new engine replicas are added, existing replicas retain ownership of all their partitions until their lease expires (TTL-based, typically 60s). New replicas may sit idle until rebalancing completes.

**Workaround:** After adding replicas, restart all engine pods at once so partitions redistribute immediately:

```bash
kubectl delete pods -l app=engine -n thyme
```

## Broker rebuild and stale offsets

If the Kafka-compatible broker cluster is rebuilt (for example, scaling from 1 to 3 brokers in a way that loses topic data), the engine's persisted progress will reference offsets that no longer exist. Workers will fail with `Broker: Offset out of range`.

**Workaround:** When rebuilding the broker cluster, also clear engine state (data volume + persisted streaming progress) before restarting the engine. See the operations runbook in [thyme-infra](https://github.com/SamuelJenkinsML/thyme-infra) for the exact procedure.
