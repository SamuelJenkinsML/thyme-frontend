Thyme runs on Kubernetes. The reference deployment targets AWS EKS with a GitOps workflow using ArgoCD, but the building blocks are standard and portable to any Kubernetes platform.

## Reference architecture

```
GitHub Actions (CI)
    │  Build multi-arch container images (AMD64 + ARM64)
    │  Push to a container registry
    ▼
ArgoCD
    │  Sync manifests on tag updates
    ▼
Kubernetes Cluster
    ├── Definition Service (port 8080)
    ├── Engine
    ├── Query Server (port 8081)
    ├── Postgres (managed or in-cluster)
    └── Kafka-compatible broker (e.g. Redpanda)
```

## Infrastructure bootstrap

The reference Terraform modules in [`thyme-infra`](https://github.com/SamuelJenkinsML/thyme-infra) provision the full stack:

1. **VPC and networking** — private subnets, NAT gateways
2. **EKS cluster** — managed Kubernetes with ARM64 (Graviton) nodes
3. **ECR repositories** — container image storage for all services
4. **ALB Ingress Controller** — HTTPS termination with ACM certificates
5. **Secrets Manager** — API keys and database credentials

```bash
cd thyme-infra/terraform
terraform init
terraform apply
```

## Kubernetes manifests

Manifests use Kustomize with a base + overlay structure:

- `manifests/base/` — shared manifests for all environments
- `manifests/overlays/demo/` — demo environment patches
- `manifests/overlays/ephemeral/` — short-lived test environments

## Quick-start scripts

```bash
# Spin up a full environment
cd thyme-infra
./scripts/spin-up.sh

# Tear down
./scripts/tear-down.sh
```

## Multi-architecture support

Container builds produce images for both `linux/amd64` and `linux/arm64`. The CI pipeline builds each architecture natively, then merges into a multi-arch manifest. ARM64 (e.g. Graviton `c7g.xlarge`) offers better price-performance for Thyme's CPU-bound aggregation workloads.

## GitHub Actions OIDC

Deployments use GitHub Actions OIDC for keyless authentication to AWS — no long-lived AWS credentials are stored in GitHub secrets.
